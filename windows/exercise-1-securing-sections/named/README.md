Write a C++ program that spawns a highly privileged process (such as a mock system service running as `SYSTEM` using tools like PsExec) and creates two named shared memory sections using `CreateFileMappingW`, where the first section explicitly passes `NULL` for the `SECURITY_ATTRIBUTES` and the second section passes a strict security descriptor generated from the SDDL string `D:(A;;GA;;;SY)`. Once your privileged process is running and holding the handles open, write a second unprivileged C++ application that attempts to open both sections using the `OpenFileMappingW` API while requesting `FILE_MAP_WRITE` access, and observe how the kernel's Security Reference Monitor happily grants you a writable handle to the `NULL` attribute section based on the default token DACL, but violently rejects your access to the strictly defined section with an `ERROR_ACCESS_DENIED` code.
